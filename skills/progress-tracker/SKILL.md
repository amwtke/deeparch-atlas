# atlas / progress-tracker

## 职责

PROGRESS.md 是 atlas 套件的**单一事实来源(single source of truth)**。本 skill 负责:

1. **初始化** PROGRESS.md(Discovery 之后,主流程开始时)
2. **更新** 进度状态机(每阶段开始/结束时)
3. **追加** 用户疑问审计记录(用户追问/换角度时)
4. **读取** 进度状态(中断恢复时)

---

## PROGRESS.md 模板

```markdown
# <主题> Atlas Progress

## 元数据
- **主题**: <主题原文>
- **短名**: <短名>          ← 目录名用,≤20 字符
- **工作目录**: atlas-output/<短名>-<yyyymmdd>/
- **创建时间**: <ISO timestamp>
- **上次更新**: <ISO timestamp>
- **当前阶段**: <stage name>

## 灵魂问题(Discovery 收集)
> "<用户的具体问题>"

## 进度状态机
- [ ] 0. Discovery
- [ ] 1. What
- [ ] 2. Why(含约束清单建立)
- [ ] 3. How
- [ ] 4. Origin
- [ ] 5. 分水岭决定
- [ ] 6. Deep
- [ ] 7. Comparison(可选)
- [ ] 8. Synthesis
- [ ] 9. 导出格式询问
- [ ] 10. 融合输出

## 约束清单(Why 阶段建立,后续阶段引用)
| # | 约束 | 来源 | 不可再分性 |
|---|------|------|-----------|
| (Why 阶段写入) | | | |

## 用户疑问/追问审计

### What 阶段
- (无) / (列出每次「追问」「换角度」的具体内容和 Claude 的回应)

### Why 阶段
...

(各阶段都有独立小节,即使无内容也保留标题)

## 选定的对比对象(Comparison Hook 后填入)
- (无) / (用户选了哪几个)

## 选定的导出格式(导出阶段填入)
- (待定) / MD / MD+视频脚本 / MD+HTML
```

## 操作规范

### 操作 1: 初始化(Discovery 完成后)

由 discovery 子 skill 在收集完灵魂问题后调用本 skill。

执行步骤:
1. 创建 `atlas-output/<短名>-<yyyymmdd>/PROGRESS.md`(`<短名>` 和 `<yyyymmdd>` 由入口 SKILL.md 派生并传入,见入口的「启动流程」)
2. 填入元数据(创建时间 = 现在,当前阶段 = "What";元数据中 **主题** 字段保存原始主题全名,**短名** 字段保存目录用短名)
3. 填入灵魂问题(从 discovery 传入)
4. 进度状态机:`[x] 0. Discovery` 其余 `[ ]`
5. 约束清单留空(等 Why 阶段填)
6. 各阶段疑问审计小节都写"(无)"占位

### 操作 2: 阶段开始(每个 stage-X 开始时)

执行步骤:
1. 读取 PROGRESS.md
2. 更新「当前阶段」字段
3. 更新「上次更新」时间戳
4. 写回 PROGRESS.md

### 操作 3: 阶段完成(每个 stage-X 结束时,用户选「懂了」之后)

执行步骤:
1. 读取 PROGRESS.md
2. 进度状态机对应行从 `[ ]` 改为 `[x]`
3. 更新「上次更新」时间戳
4. 如果是 Why 阶段,把 Why 子 skill 产出的约束清单写入 PROGRESS.md 对应表格
5. 如果是 Comparison 阶段,把用户选中的对比对象写入「选定的对比对象」小节
6. 写回 PROGRESS.md

### 操作 4: 用户追问 / 新视角 / 不懂反馈 / 产物迭代时(由 user-pacing 协议触发)

执行步骤:
1. 读取 PROGRESS.md
2. 在当前阶段的疑问审计小节追加一行,前缀按类型选:
   - `追问`:`- 第 N 次追问: <用户问题>` + Claude 回应摘要(50 字内) + `→ 已 patch 文件:<具体段落/位置>`
   - `新视角`:`- 第 N 次新视角: <用户视角>` + Claude 响应摘要 + `→ 已 patch 文件:...`
   - `不懂反馈`:`- 第 N 次不懂反馈: <用户具体说哪里不懂>` + `→ 已局部重写:<段落>`
   - `产物迭代`:`- 第 N 次产物迭代: <修订摘要>` + `→ 触发原因:...`
3. 更新「上次更新」时间戳
4. 写回 PROGRESS.md
5. **静默执行** —— 对话中不提"已记录到 PROGRESS.md"

### 操作 5: 中断恢复(用户重新启动 atlas 时)

由入口 SKILL.md 检测到 PROGRESS.md 存在时调用本 skill。

执行步骤:
1. 读取 PROGRESS.md
2. 在对话中向用户输出:
   ```
   检测到上次进度:
   - 主题: <主题>
   - 灵魂问题: "<...>"
   - 上次走到: <当前阶段>
   - 已完成: <列出已完成的阶段>
   - 上次活动时间: <时间>
   
   是否继续?
   [继续:从 <当前阶段> 开始]
   [重新开始:删除整个 atlas-output/<短名>-<yyyymmdd>/ 目录,以今天日期重建]
   [换主题:取消恢复,等待新指令]
   ```
3. 用 `ask_user_input_v0` 给三个按钮
4. 根据用户选择:
   - 「继续」→ 调度对应阶段子 skill(沿用旧目录,日期不变)
   - 「重新开始」→ 删除旧的 `atlas-output/<短名>-<yyyymmdd>/`,以**今天日期**新建,调度 discovery
   - 「换主题」→ 退出,等待用户给新指令

---

## 注意事项

- **绝不在阶段开始/结束/产物迭代时跳过本 skill** —— 这是套件的灵魂机制
- **完全静默执行** —— 对话中**绝不**说"PROGRESS.md 已更新 / 进度已记录 / 当前阶段标记为 X"等管道术语;用户不应该知道 PROGRESS.md 的存在(除非他主动问)
- **PROGRESS.md 的格式必须严格遵循模板** —— 中断恢复时的解析依赖固定结构
- **约束清单的编号 C1, C2, ... 必须保持稳定** —— 一旦 Why 阶段写入,后续阶段引用编号才有意义,新增约束往末尾追加,不重排
- 写入 PROGRESS.md 失败时,**仍然静默** —— 教学不受影响,只是失去恢复能力,告知用户为时未晚(可在用户最终关闭对话或主动问起恢复时再说)

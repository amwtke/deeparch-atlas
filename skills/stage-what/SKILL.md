# atlas / stage-what

## 阶段目标

让用户能用一句话跟别人说清楚 <主题> 是什么。

**克制原则**:What 阶段不深入机制、不讲源码、不讲历史——那些是后续阶段的事。这里只建立"骨架认知"。

---

## 讲解结构(150~250 行)

输出到 `stages/01-what.md`,结构如下:

```markdown
# 阶段 1:<主题> 是什么

## 一句话定义
[< 40 字的定义,用户读完就能复述]

要求:
- 严格 < 40 字
- 禁止套话("业界领先"、"高性能"等)
- 禁止同义反复("分配器是用来分配内存的")
- 写不出来就回去重想——做不到一句话定义,说明还没真正理解

## 跨领域类比
[一个日常生活/自然现象/历史故事的类比]

要求:
- 必须跨领域(快递包裹、河道水流、图书馆借阅卡等)
- 禁止"就像 Redis 的 X" 这种同行业类比(读者可能不知道 Redis)
- 好的类比让读者在类比内部就能推导出主题的关键性质

## 一张全景图
[ASCII art 或文字描述,画出这个技术的"外形"]

要求:
- 让读者看完就能指着说"哦,这里是 A,这里是 B"
- 标出主要部分和它们的边界
- 不深入内部机制(How 阶段的事)

## 关键词速查表

| 术语 | 一句话说明 |
|------|-----------|
| ... | ... |

要求:
- 5~8 个术语,后续阶段会反复出现
- 每个一句话,不展开

## 它在系统/技术栈里的位置
[这个技术上面是谁、下面是谁、平行的有谁]

要求:
- 用一张简单的层级图或文字描述
- 让读者知道它的"邻居"

## 呼应灵魂问题
[一段话,说明 What 阶段的内容如何为回答灵魂问题打下基础]

例如灵魂问题是"chunk 为什么是 16B 不是 8B":
> "我们刚刚画出了 chunk 的全景——它是一个内存包裹,有 header 和 user data 两段。
> 你的灵魂问题问的是 header 的具体大小,这个在 How 和 Deep 阶段会精确回答。
> 现在记住:chunk 这个抽象本身,是为了解决我们下一阶段(Why)要讲的核心约束。"
```

## 执行流程

### 第一步:静默标记阶段开始

调用 `skills/progress-tracker/SKILL.md` 的「阶段开始」操作,标记当前阶段为 "What"。**对话中不提此事。**

### 第二步:**先问 3~5 个对齐问题**(必做,不可跳过)

按入口 SKILL.md 「Stage 开场对齐纪律」,在生成产物前必须先问用户 3~5 个对齐问题,确认本阶段的范围 / 特点 / 目的 / 口径 / 需求。

What 阶段典型问题(临场调整):

1. **聚焦深度**:你期望 What 阶段聚焦在"是什么 + 长什么样",还是顺带扯边界("它**不是**什么")?
2. **类比维度**:你希望从哪个领域类比?(物理 / 数学 / 日常 / 其他工程系统)
3. **全景图深度**:概念图(只画"是个什么形状")还是结构图(展开主要部件)?
4. **用户背景**:你已经知道相关的什么概念?需要补 / 避开重复哪些?
5. **(可选)关键词偏好**:有没有几个词你已经听过但没搞清楚,本节顺手澄清?

用户答完(包括"你看着办"+ Claude 列推断让用户点头)再进入第三步。

### 第三步:写第一稿到 stages/01-what.md

按上面结构,生成 What 阶段的初稿。

**关键要求**:
- 不深入机制(那是 How / Deep 的事)
- 不讲历史(那是 Origin 的事)
- 不讲对比(那是 Comparison 的事)
- 内容必须呼应灵魂问题(在最后一段说明)
- **克制原则硬要求**:除非 <主题> 本身就是某个内部组件(如"arena"),否则**绝不在 What 里展开内部组件**。例如讲 malloc,不要在 What 里展开 arena/bin/top_chunk —— 那会把"骨架认知"逼成"机制图谱",违反 What 阶段的本分

### 第四步:在对话中**自然地**讲清楚要点 + 抛反问

**不要说**"What 阶段已写入 stages/01-what.md(共 XX 行)"。这是管道术语,违反对话语言纪律。

改说(示例,临场调整,核心是**自然 + 留路径让用户找到产物**):

```
我先把 <主题> 的轮廓画了一下,草稿放在 atlas-output/.../stages/01-what.md。
要点其实就三件事:
- 它是<一句话定义>
- 类比成<跨领域类比的精髓>
- 它的"邻居"长这样:<上下游 + 平行 implementations>

不过我不想急着往下走 —— 想先拷问一下这个轮廓:<反问>
```

**反问要贴用户当前理解的边缘**,例如:

- "你觉得这个类比里最关键的一点是哪个?是'复用'还是'多分店'?"
- "如果让你只用一个词描述它,你会用哪个?"
- "你觉得它跟 <某邻居> 最大的区别是什么?"

抛完反问就**等用户回应**,进入 user-pacing 协议(skills/three-options/SKILL.md)。

### 第五步:进入 user-pacing 协议

按 `skills/three-options/SKILL.md` 协议处理用户回应:

- **追问 / 新视角 / 不懂反馈** → 答清楚 + **patch `atlas-output/<短名>-<yyyymmdd>/stages/01-what.md`** + 抛新反问
- **附和(嗯/好/明白)** → **不要推进**,再抛反问或留白
- **明确推进信号**(继续 / 走吧 / 下一个 / 这块够了 / 我懂了接着讲) → 加载 `skills/stage-why/SKILL.md`,**用自然过渡语**接到 Why,例如:
  - "那顺着这个,我想跟你回到一个很基础的问题 —— 没有这个东西的世界会怎样?"
  - "顺着你刚才说的 X,我下一步想拿一份'约束清单'去拷问它的存在必要性。"

**绝不说**"现在进入 Why 阶段 / 调度 stage-why"。

---

## 注意事项

- **一句话定义是 What 阶段的灵魂**——超过 40 字就回去重想
- **类比禁止同行业**——技术读者可能比你想的更"小白"
- **不深入**——遇到想展开的细节,克制住,告诉自己"How 阶段会讲"
- 阶段文件命名严格 `01-what.md`(小写、带数字前缀)
- **stage 文件是活产物** —— 在用户明确推进之前会被反复 patch(用户追问 / 新视角 / 不懂反馈触发);见 three-options 的「产物迭代规则」

---

## 反模式

1. **一句话定义写超过 40 字** —— 没真正理解,回去重想
2. **类比用"就像 Redis 的 X"** —— 读者可能不知道 Redis
3. **What 阶段塞满细节(展开 arena / bin / top_chunk 等内部组件)** —— 违反克制原则,用户接不住
4. **不写"呼应灵魂问题"段落** —— 失去与 Discovery 的衔接
5. **没有明确推进信号就推进到 stage-why** —— 违反 gatekeeper 契约
6. **在对话中说"What 阶段已写入..." / "调用 three-options" / "现在进入 Why 阶段"** —— 暴露管道术语,违反对话语言纪律
7. **追问只在对话里答,不 patch stage 文件** —— stage 文件就废了,不再反映用户当前理解

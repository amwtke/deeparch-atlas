# atlas / stage-why

## 阶段目标

让用户理解"没有这个技术的世界会怎样"+ 它面对的**不可再分约束**。

**关键产出**:《约束清单》(C1, C2, ...)。这是 atlas 的脊梁,**所有后续阶段都要交叉引用**。

---

## 讲解结构(200~300 行)

输出到 `stages/02-why.md`:

```markdown
# 阶段 2:<主题> 为什么必须存在

## 没有它的世界
[2~3 个具体场景,描述工程师在没有这个技术时的具体痛苦]

要求:
- 越具体越好(具体到代码、数字、调用链)
- 不要写抽象痛点("性能差")
- 这是 Why 的灵魂——读者如果不理解"没有它会怎样",就理解不了"它凭什么值得"

## 它面对的不可再分约束(《约束清单》)
| # | 约束 | 来源 | 不可再分性 |
|---|------|------|-----------|
| C1 | ... | ... | ... |
| C2 | ... | ... | ... |
| ... | ... | ... | ... |

要求:
- 至少 5 条,通常 5~8 条
- "不可再分"意味着不能再拆——"高性能"不是约束,"内核 syscall 微秒级"才是
- "来源"标明:应用语义 / OS 设计 / CPU 架构 / 并发模型 / ABI 历史 / 经验事实
- **编号 C1, C2 必须稳定**——一旦写入,后续阶段引用编号

## 它的核心 insight
[1~2 个能让用户"啊哈"一下的关键想法]

要求:
- 必须能被用户复述
- 写成"用 X 换 Y"或"把 X 转化为 Y"的形式更容易记
- 例:"用 16 字节换 O(1)"、"把查表问题转化为反向偏移问题"

## 这个 insight 怎么解决了哪些约束
[把 insight 对应到约束清单的哪些条]

例如:
> "chunk 内联 header 的 insight,精确解决了 C5(信息量约束)+ C1(纳秒级响应)的张力。
>  代价是放弃了某些 C3 的优化空间(cache 友好性),但这个代价在 1996 年的硬件上是可接受的。"

## 呼应灵魂问题
[一段话,说明 Why 阶段的约束清单如何为回答灵魂问题铺平道路]

例如灵魂问题是"chunk 为什么是 16B 不是 8B":
> "你的灵魂问题问 16B 这个具体数字。这个问题的答案藏在 C3(对齐 + cache line)和
>  C5(prev_size + size 两个 8B 字段必不可少)的交集里。等到 How 阶段你看到 header
>  的字段布局,会自然得到这个数字。"
```

## 执行流程

1. **静默标记阶段开始**(调用 progress-tracker,对话不提)
2. **生成 stages/02-why.md** 第一稿,按上面结构
3. **静默把约束清单写入 PROGRESS.md**(中央存放,后续阶段查询)
4. **在对话中自然地讲清楚 + 抛反问**(不说"Why 阶段已写入..."):
   ```
   顺着你刚才说的 <衔接 What 阶段对话>,我把"没有它的世界"草稿放在了
   atlas-output/.../stages/02-why.md。

   核心是这几条不可再分的约束:C1 ... / C2 ... / ...(<3-5 条简述>)
   它们合起来逼出了 <主题> 这个东西非存在不可的形状。

   不过先拷问一下:<反问,贴用户当前理解>
   ```
   反问示例:
   - "你觉得这几条约束里,哪一条是最'硬'的?哪一条其实是软约束?"
   - "如果让你只保留一条,只用它就要逼出 <主题>,你会留哪条?"
   - "你能想到哪个反例 —— 某个场景这条约束其实不成立?"
5. **进入 user-pacing 协议**(skills/three-options/SKILL.md):
   - 追问 / 新视角 / 不懂 → 答 + **patch stages/02-why.md** + 抛新反问
   - 附和 → 不推进,再抛反问
   - 明确推进信号 → 加载 `skills/stage-how/SKILL.md`,自然过渡
   - **约束编号 C1, C2, ... 永久固定**:即便用户要求重写产物,编号不重排;新约束往末尾追加 C(N+1)

## 关键约束(本阶段独有)

### 约束清单的稳定性

一旦本阶段把约束清单写入 PROGRESS.md,**编号 C1, C2, ... 永久固定**。

如果用户在后续阶段(如 Deep)发现需要补充约束,通过 progress-tracker 在末尾追加 C(N+1),**不重新编号**。

### 约束 vs 功能

约束清单的常见错误是写成"功能列表":
- ❌ "高性能" → 这是功能不是约束
- ✅ "syscall 跨态边界微秒级" → 这是物理约束
- ❌ "易用性" → 这是功能
- ✅ "C ABI 的 free(p) 只有指针" → 这是约束

如果某条写不出"不可再分性"那一栏,说明它不够"基础",再往下钻一层。

---

## 反模式

1. **约束清单写成功能列表** —— "高性能"不是约束,"内核 syscall 微秒级"才是
2. **不写《呼应灵魂问题》段落** —— 失去与灵魂问题的衔接
3. **没有它的世界写得抽象** —— 必须具体到场景、代码、数字
4. **约束编号在后续阶段被重新分配** —— C1, C2 是脊梁,稳定性优先
5. **不调用 progress-tracker 把约束清单写入 PROGRESS.md** —— 后续阶段无法交叉引用

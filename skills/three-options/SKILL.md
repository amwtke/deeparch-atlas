# atlas / user-pacing(文件名 three-options 是历史遗留)

## 职责

每段教学讲完后,本 skill 协调"留在当前话题 vs 推进"。

**核心契约**:

- Claude **永远不主动推进**,默认留在当前话题
- 推进的唯一触发器:用户的**明确推进信号**(白名单见下)
- 用户的所有非推进回应(追问 / 新视角 / 不懂 / 附和),都视作"对当前 stage 产物的迭代输入"
- 必须把对话里出现的新理解、追问的解答、新视角,**patch 回 `stages/0X-stage.md`**(surgical edit,不擦写)

绝不在对话中说"我现在调用 three-options 给你三个选项",**没有"按钮 widget" 这回事** —— 这是个对话过程。

---

## 对话节拍

每段教学讲完后,Claude:

1. **抛一个反问**(检验用户接住的程度,贴当前理解的边缘 —— 例:"你觉得 free 为什么不传 size?")
2. **等用户回应**
3. **分类并响应**(见下)
4. **回到第 2 步**,直到收到明确推进信号

如果反问 2~3 轮用户都只是附和(嗯/好/明白)、没有新疑问也没有推进词,可以**主动把球抛回去**:"这块还有想拧紧的吗?还是可以接着往下走了?" —— 这是把推进决定权显式交给用户,**不是替用户决定**。

---

## 用户回应分类

### A. 明确推进信号(放行)

**白名单**(必须严格匹配某种意图):

- "继续 / 接着讲 / 走吧 / 下一个 / 下一阶段 / 进下一个"
- "这块差不多了 / 这块够了 / 可以走了 / 可以推进 / 没问题了"
- "ok 推进 / OK 继续 / 嗯走吧 / 好了往下"
- "我懂了,接着讲 / 我懂了,继续"(注意:**单纯的"我懂了 / 嗯 / 好 / 明白"不算** —— 那是附和)

判断要严格。**模糊就当作附和,继续等**。宁可多问一句"还是可以走了?",不要替用户推进。

收到明确推进信号后(**严格按以下顺序**):

1. 调用 progress-tracker 标记本阶段完成
2. **静默封存当前 stage 的产物**(git snapshot,见入口 SKILL.md 的「产物封存协议」):
   ```bash
   cd <repo root>
   git add atlas-output/<短名>-<yyyymmdd>/
   git commit -m "atlas(<短名>): finish <stage 名> stage

   <可选:一句话总结本轮用户接住的核心 / 笔记关键修订>"
   ```
   - 如果工作树没有未 stage 的改动(理论上不该发生,但保险起见),`git diff --cached --quiet` 检测,空 commit 跳过
   - git 失败时,在对话中告知用户("封存这一步出了问题:<错误摘要>。教学可以继续,但建议结束前手动 git 一下");**不阻塞推进**
3. 加载调用方告知的"下一阶段子 skill"
4. **过渡语自然化**(不说"现在进入 Why 阶段",**也不说"已 commit"**):
   - 例:"那顺着这个,下一个我想跟你回到原点问 —— 没有 malloc 的世界会怎样?"
   - 例:"接下来想拿几个具体场景拷问一下你刚才形成的理解。"

### B. 追问 / 疑问

**特征**:带问号的句子,或"为什么 X / X 是什么 / 怎么 X / X 没看懂 / 再讲讲 X"。

**响应**:

1. 在对话里答清楚
2. **把答复要点 patch 回 `stages/0X-stage.md`**:
   - 如果答复修正了原文某段 → surgical edit 那一段
   - 如果答复新增了视角 → 在合适位置 insert,或在文末「对话凝固」小节追加
3. 静默调用 progress-tracker 记一笔(操作:用户追问)
4. 抛新反问或留白,等用户

### C. 新视角 / 推断 / 不同意

**特征**:"我觉得 / 我理解 / 是不是其实是 X / 这跟 Y 不像吗 / 那如果 Z 呢 / 我想换个角度看"。

**响应**:

1. 在对话里回应(认同 / 修正 / 引申)
2. **把这个新视角融入 `stages/0X-stage.md`**(append 一个用户视角小节,或改某段表述)
3. 静默调用 progress-tracker 记一笔(操作:用户视角)
4. 抛新反问或留白

### D. 反馈"讲得不好 / 跳跃 / 过快 / 太密"

**特征**:"我觉得有点跳 / 这里没听懂 / 概念太多 / 太快 / 你讲得不清楚 / 这个我不懂"。

**响应**:

1. 在对话里**承认问题**,定位具体哪里跳 / 哪个概念过于密集
2. **重写 `stages/0X-stage.md` 的相关部分**(局部重写,不是整体擦写,除非用户明确说"全部重来")
3. 在对话里讲清楚那块,再抛反问验证
4. 静默调用 progress-tracker 记一笔(操作:产物迭代 / 局部重写)

### E. 附和 / 无信号

**特征**:"嗯 / 好 / 明白 / 对 / 知道了 / 哦"(**没有**"继续/下一个/走吧"等推进词)。

**响应**:

- **不要推进**
- 再抛一个不同角度的反问,或留白等用户主动说话
- 反问 2~3 轮后用户仍只附和 → 主动询问 "这块还有想拧紧的吗,还是可以接着往下了?"(把球抛回)

## 产物迭代规则

`stages/0X-stage.md` **是活产物**,在用户给出明确推进信号之前会被反复 edit。

- **patch 优于擦写**。只在用户明确说"全部重来 / 换个角度从头讲"时整体重写
- 每次 patch 在文件末尾的「修订记录」小节追加一行:`yyyy-mm-dd HH:MM | 修订摘要 | 触发原因(用户追问/新视角/不懂反馈)`
- 「修订记录」小节如果不存在则创建
- patch 前可以快速读取当前文件(用 Read 工具)以避免冲突

---

## 调用方契约

stage skill 用本协议时,要在自己的执行流程里明确:

1. **当前 stage 文件路径**(如 `atlas-output/<短名>-<yyyymmdd>/stages/01-what.md`) —— 用于 patch
2. **下一阶段子 skill 路径**(如 `skills/stage-why/SKILL.md`) —— 用户明确推进时加载
3. **本阶段反问素材**(可选) —— 反问应贴当前内容的边缘,由 stage skill 临场生成最贴

**不需要传"三个备选切入点"** —— 旧版的"换角度三选"已废弃。如果用户要换角度,他会自己说("从接口反推 / 用类比讲讲 / 不要那么跳"),由当前 stage skill 临场处理。

---

## 反模式

1. **Claude 主动推进**(无明确推进信号就加载下一阶段) —— 违反核心契约
2. **把"嗯/好/明白"当推进信号** —— 那是附和,不是推进
3. **追问/新视角只在对话里答,不 patch stage 文件** —— 文件就废了,不再反映用户当前理解
4. **每次 patch 都整体擦写** —— 应该 surgical 修改,保留已有内容
5. **在对话中说"我现在给你三个选项 / 调用 three-options"** —— 暴露 skill 内部术语,违反对话语言纪律
6. **抛反问后用户附和 1 次就主动推进** —— 至少给 2~3 个对话回合;真要确认时把球抛回给用户
7. **过渡到下一阶段时说"现在进入 X 阶段"** —— 用自然语言连贯过渡,见入口 SKILL.md 的「对话语言纪律」

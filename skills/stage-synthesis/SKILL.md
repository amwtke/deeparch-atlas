# atlas / stage-synthesis

## 阶段目标

把整个学习过程**提炼成可推广的方法论**——让用户拿这套方法分析其他系统(epoll, MVCC, Raft, TCP...)。

**Synthesis 是 atlas 的灵魂收束**。如果 Synthesis 只是前文摘要,整个 atlas 流程就失败了——它必须有**新的产出**:
- 五元组表(或三元组,看模式)
- 同构系统(2~3 个其他领域的同构例子)
- 局部最优论证
- 方法论沉淀

---

## 两种模式

根据用户走过的阶段,Synthesis 自动选择:

### 轻量模式(用户在分水岭选了"满足导出")

- **覆盖阶段**:What + Why + How + Origin
- **不含**:Deep 的反事实小试 + Comparison 的横向对照
- **核心产物**:三元组表(决策 / 约束 / 代价)
- **行数**:~300~400 行

### 硬核模式(用户走完了 Deep,可能含 Comparison)

- **覆盖阶段**:全部
- **核心产物**:五元组表(决策 / 约束 / 代价 / 反事实候选 / 现实对照)
- **行数**:~400~600 行

## 讲解结构

输出到 `stages/07-synthesis.md`:

### 轻量模式结构

```markdown
# 阶段 7:<主题> 的设计哲学(轻量版)

## 7.1 三元组表
| 设计决策 | 解决的约束 | 付出的代价 |
|---------|-----------|-----------|
| ... | ... | ... |

要求:
- 8~12 行
- 决策从 What/Why/How 阶段提取,代价从 Why 阶段提取

## 7.2 同构系统:这套设计哲学在哪里又出现过
[2~3 个其他领域的同构例子]

## 7.3 时代局限
[当年的约束 vs 今天的约束;后续演化怎么补救]

(从 Origin 阶段的内容提取)

## 7.4 方法论沉淀(三步法)
1. 识别约束(C1, C2, ...)
2. 列举候选 + 代价
3. 看历史 / 现实里别人怎么选

## 呼应灵魂问题
[最终回答灵魂问题的精炼版本]
```

### 硬核模式结构

```markdown
# 阶段 7:<主题> 的设计哲学与方法论沉淀

## 7.1 五元组表(atlas 的核心产物)
| 设计决策 | 解决的约束 | 付出的代价 | 反事实候选 | 现实对照 |
|---------|-----------|-----------|-----------|---------|
| ... | ... | ... | ... | ... |

要求:
- 12~15 行
- 决策从全部前阶段提取
- 反事实候选从 Deep 的「反事实小试」整理
- 现实对照从 Comparison 的对照对象整理(若无 Comparison,用 Deep 反事实小试里出现的对比)

## 7.2 同构系统:这套设计哲学在哪里又出现过
[2~3 个其他领域的同构例子]

例如 chunk + boundary tag 的设计哲学,在以下场景同构:
- 文件系统的 inode + block group(元数据内联 + 边界邻接)
- TCP 包的 header + payload + checksum 尾(边界标签)
- Java 对象的 mark word + class pointer(in-use/free 状态复用)

## 7.3 局部最优论证(关键段落)
[为什么 <主题> 在它的约束组合下是局部最优]

格式:
> 给定约束 {C1, C2, ..., Cn},<主题> 是**唯一能同时满足所有约束的设计**。
> 
> - 取消 C7(允许改 ABI):候选 X 更优 → 现实里 Rust 这么做
> - 取消 C5(让分配器知道 size):候选 Y 更优 → 现实里 Go 这么做
> - 取消 C1(允许 free 慢):候选 Z 可行 → 但没人选
> 
> <主题> 不是 1996 年最聪明的设计,而是**当时约束组合下唯一过得了所有约束筛的设计**。

## 7.4 时代局限与演化
[当年的约束 vs 今天的约束;后续演化怎么补救]

(从 Origin + Comparison 整理)

## 7.5 方法论沉淀(五步法,可推广到其他系统)
1. **识别约束**:不可再分的物理/数学/工程/ABI 边界
2. **列举候选**:抽象层面的所有可能解
3. **代价分析**:每个候选违反哪些约束、付出什么代价
4. **现实对照**:历史/工业界谁选了什么(天然对照实验)
5. **局部最优证明**:为什么是这个候选而不是其他

这五步可以推广到任何系统分析:
- epoll vs kqueue vs IOCP
- MVCC vs 2PL
- Raft vs Paxos
- TCP vs QUIC
- ...

## 呼应灵魂问题
[最终回答灵魂问题,基于全文所有阶段的洞察]
```

## 执行流程

1. **调用 progress-tracker** 标记阶段开始为 "Synthesis"
2. **判断模式**:
   - PROGRESS.md 中是否标记了 Deep 阶段完成?
     - ❌ → 轻量模式
     - ✅ → 硬核模式
3. **从 PROGRESS.md 读取**:约束清单 + 灵魂问题 + 用户在 Comparison 选了什么
4. **从各 stage.md 提取关键内容**:
   - 设计决策(从 How / Deep)
   - 反事实候选(从 Deep 的反事实小试)
   - 现实对照(从 Comparison,若无则从 Deep 反事实小试)
   - 时代局限(从 Origin)
5. **先问 3~5 个对齐问题**(必做,不可跳过 —— 见入口 SKILL.md「Stage 开场对齐纪律」)

   Synthesis 是收束阶段,对齐决定"谁是必保的核心、谁可以省略"。典型问题(临场调整):

   1. **收束目的**:复习 / 教学他人 / 自己的 long-form 笔记 / 给团队分享?(决定语气和详细度)
   2. **必保的洞察**:走完前几个 stage,你最不希望丢掉的是哪几条?(让用户先报,Claude 后面对照确保都收录)
   3. **方法论沉淀的泛化**:把"约束 → 决策 → 反事实"这套泛化到其他系统的同构,你想拿它去看哪个系统?(用作 §7.2「同构系统」段落的具体例子)
   4. **长度偏好**:精炼版(~3000 字)还是详尽版(~8000 字)?
   5. **(可选)用户贡献的元规则收录**:走过程中你贡献的元洞察(如 atlas malloc 案例的"约束反向演化"+ "约束复合分解")要不要单独成节?

   用户答完(包括"你看着办"+ Claude 列推断让用户点头)再进入第六步。

6. **生成 stages/07-synthesis.md** 按对应模式结构(根据用户答的对齐口径决定核心保哪些 / 长度)
7. **在对话中自然地讲清楚收束的产物 + 抛反问**(不说"Synthesis 阶段已写入..."):
   ```
   到这里,我把整个讨论收束成了一份方法论沉淀,放在 atlas-output/.../stages/07-synthesis.md。
   核心产物:<三元组表 / 五元组表>(<X 行>),配 <三步法 / 五步法> 的方法论提炼。

   先拷问你一个收束级的反问:<例如"如果让你拿这个方法论去看另一个看似无关的系统,你会拿它去看哪个?">
   ```
8. **进入 user-pacing 协议**(skills/three-options/SKILL.md):
   - 追问 / 新视角 → 答 + **patch stages/07-synthesis.md** + 抛新反问
   - 明确推进信号(继续 / 收尾 / 导出) → 加载 `skills/final-export/SKILL.md`,自然过渡到导出格式询问

   注意:Synthesis 之前老版"不调用 three-options 直接 export"已废弃 —— 收束之后用户也应该有机会拷问、补充、修正方法论沉淀。

## 关键约束

### 必须有"新产出"

Synthesis 失败的标志是"只是前文摘要"。必须有:

- **五元组(或三元组)表**:这是从所有阶段抽取的高度浓缩
- **同构系统**:跳出 <主题> 看其他领域的同构,这是真正的"可推广"
- **局部最优论证**:这是第一性原理的最终落地
- **方法论沉淀**:让读者拿走方法,不是拿走知识

如果以上四个产物有任何一个写成"前文已经讲过的内容",Synthesis 失败。

### 五元组表的"现实对照"列

如果用户走完了 Comparison,直接用 Comparison 的对照对象。
如果用户跳过了 Comparison,从 Deep 的反事实小试中提取——那里其实已经有"现实里有没有人这么做"的信息。

---

## 反模式

1. **Synthesis 写成前文摘要** —— 没有新产出,Synthesis 失败
2. **三元组/五元组表行数太少**(< 8) —— 拆得不够细
3. **行数太多**(> 15) —— 没分主次
4. **代价列写"几乎没有"** —— 任何决策都有代价,写不出说明分析不到位
5. **同构系统列举不准** —— 不是真同构而是表面相似
6. **方法论沉淀没有"可推广性"** —— 只能用于 <主题>,不能推广就是知识不是方法
7. **不呼应灵魂问题** —— 整个流程的目标问题在收束时缺席

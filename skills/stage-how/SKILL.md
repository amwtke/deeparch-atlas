# atlas / stage-how

## 阶段目标

让用户对 <主题> 内部机制有**骨架级理解**——能讲清楚"它内部大致怎么分工 + 一次完整的工作流程"。

**克制原则**:How 阶段不深入源码、不剖析每个函数——那是 Deep 阶段的事。这里只到"骨架"。

---

## 讲解结构(250~350 行)

输出到 `stages/03-how.md`:

```markdown
# 阶段 3:<主题> 大致怎么工作

## 核心机制骨架
[3~5 步讲清楚内部大致怎么工作]

要求:
- 用"首先 → 然后 → 接着 → 最后"连起来
- 每步 1~2 句,不展开
- 遇到想深入的细节,克制住,告诉自己"Deep 阶段会讲"

## 一个最小 demo 或伪代码
[30~80 行,只为让用户获得"手感"]

要求:
- 可运行 / 可读懂的伪代码
- 标记关键观察点(用注释说明"这一行会触发 X")
- 不剖析,只展示"输入是什么、输出是什么、触发了什么"

## 一张对比图
[有它 vs 没它,或它 vs 常见替代方案]

要求:
- ASCII 表格或文字对照
- 让读者一眼看出"差别在哪"

## 三个常见误解
[戳破读者最容易产生的误解]

要求:
- "你可能以为 X,其实 Y" 的句式
- 必须是真实容易出现的误解,不是凑数

## 《约束回扣》
[How 阶段呈现的机制,主要解决了约束清单的哪几条]

要求:
- 用约束编号交叉引用(C1, C5 等)
- 一段话(100~200 字)
- 这是第一性原理内核的关键渗透点

## 呼应灵魂问题
[一段话,说明 How 阶段的骨架是否已经能回答灵魂问题]

如果灵魂问题在 How 阶段就能完整回答,直接给答案;
如果还需要更深入,说明"完整答案在 Deep 阶段"。
```

---

## 执行流程

1. **静默标记阶段开始**(调用 progress-tracker,对话不提)
2. **从 PROGRESS.md 读取约束清单**(本阶段需要交叉引用)
3. **生成 stages/03-how.md** 第一稿,按上面结构
4. **在对话中自然地讲清楚 + 抛反问**(不说"How 阶段已写入..."):
   ```
   接着你的问题,我把核心机制的骨架画在了 atlas-output/.../stages/03-how.md。
   3~5 步:<复述骨架>。这套骨架对应到 Why 阶段的 C1 + C5 + C7。

   先拷问一下:<反问>
   ```
   反问示例:
   - "你觉得哪一步如果省掉,整个机制就崩了?"
   - "为什么是这个顺序,反过来跑会怎样?"
5. **进入 user-pacing 协议**(skills/three-options/SKILL.md):
   - 追问 / 新视角 / 不懂 → 答 + **patch stages/03-how.md** + 抛新反问
   - 明确推进信号 → 加载 `skills/stage-origin/SKILL.md`,自然过渡

---

## 注意事项

- **克制深入** —— 这是 How 与 Deep 分工的红线。遇到想展开的细节,用一句"Deep 阶段会精确剖析"过渡
- **demo 不要追求完整** —— 30~80 行,只为手感,不要写产品级代码
- **三个误解不是 FAQ** —— 是戳破读者**会产生**的错觉,不是回答用户问题
- **约束回扣必须用约束编号** —— C1, C5 而不是"性能约束"

---

## 反模式

1. **塞入源码细节** —— 那是 Deep 的事
2. **demo 写成完整产品** —— 30~80 行就够,不要超
3. **三个误解写成 FAQ** —— FAQ 是回答问题,误解是戳破错觉
4. **不交叉引用约束编号** —— 散了脊梁
5. **不呼应灵魂问题** —— 失去衔接

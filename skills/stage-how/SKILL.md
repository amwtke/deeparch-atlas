# atlas / stage-how

## 阶段目标

让用户对 <主题> 内部机制有**骨架级理解**——能讲清楚"它内部大致怎么分工 + 一次完整的工作流程"。

**克制原则**:How 阶段不深入源码、不剖析每个函数——那是 Deep 阶段的事。这里只到"骨架"。

---

## 讲解结构

输出到 `stages/03-how.md`,按入口 SKILL.md 的「Stage 产物结构纲领」组织(顶部速查 → §0 三件事 → §1 概览图 → §2~§N 细节 → 末尾闭环 + 修订记录)。

```markdown

   #### C2 — ...

## §0 从 Why 走到 How:三件事记住    ← How 的「三件事」 = ~3 个核心组件
   ### §0.1 <第 1 个组件,典型如 chunk> — 从 Cx + Cy 推出
```

### How 三件事的标准选法

How 阶段的「三件事」**就是 ~3 个最核心的组件**(从 Why 阶段约束推出)。各主题示例:

| 主题 | 三件事 | 各自对应的 Cn |
|------|-------|-------------|
| **glibc malloc** | chunk(物理实体) / bin(空闲索引) / arena(容器) | C5+C6 / C1+C6 / C7+C2 |
| **B+ 树** | 节点(物理) / 索引层(逻辑) / 有序链表(范围扫描) | 取决于 Why 阶段约束 |
| **io_uring** | SQ(提交队列) / CQ(完成队列) / 共享内存映射 | ... |

**纪律**:三件事**只引 3 个**,不是 5 个不是 7 个;有更多组件,留给 §2~§7 展开。三件事一旦立住,后面细节就有挂载点。

### 克制原则(How vs Deep 红线)

- **How**:讲到"骨架"为止 —— 三件事 + 它们怎么协作 + 5 步流程
- **Deep**:精确数字 + 反事实小试 —— "为什么 fastbin 上限是 64 字节"、"为什么 chunk header 是 16 字节"
- 遇到想展开数字的具体推导,**克制住**,用"Deep 阶段会精确剖析"过渡

### demo 纪律

- 30~80 行伪代码,不要写产品级
- 每一行可以指着说"这一行对应 Cn 或第 X 件事"
- 不展开边界条件 / 安全检查 / 性能优化(都属于 Deep 范畴)

## 执行流程

1. **静默标记阶段开始**(调用 progress-tracker,对话不提)
2. **从 PROGRESS.md 读取约束清单**(本阶段需要交叉引用)
3. **先问 3~5 个对齐问题**(必做,不可跳过 —— 见入口 SKILL.md「Stage 开场对齐纪律」)

   How 阶段典型问题(临场调整):

   1. **三件事(核心组件)清单**:你已经有印象哪几个组件是核心?先报一下,Claude 把"用户视角"和"Claude 推断"并列展开
   2. **端到端流程**:1 个场景(典型 alloc/free)够,还是 2~3 个场景(覆盖边界 / 异常 / 多线程并发)?
   3. **概览图风格**:抽象 3 层嵌套(只看结构),还是带具体例子(看物理 vs 逻辑视图)?
   4. **常见误解**:你之前对这个主题有过哪些"以为是 X,其实 Y"的踩坑?Claude 优先把这些列入 §7
   5. **(可选)伪代码深度**:30~40 行高度简化,还是 80 行带注释贴近源码?

   用户答完(包括"你看着办"+ Claude 列推断让用户点头)再进入第四步。

4. **生成 stages/03-how.md** 第一稿,按上面结构(根据用户答的对齐口径决定深度/广度)
5. **在对话中自然地讲清楚 + 抛反问**(不说"How 阶段已写入..."):
   ```
   接着你的问题,我把核心机制的骨架画在了 atlas-output/.../stages/03-how.md。
   3~5 步:<复述骨架>。这套骨架对应到 Why 阶段的 C1 + C5 + C7。

   先拷问一下:<反问>
   ```
   反问示例:
   - "你觉得哪一步如果省掉,整个机制就崩了?"
   - "为什么是这个顺序,反过来跑会怎样?"
6. **进入 user-pacing 协议**(skills/three-options/SKILL.md):
   - 追问 / 新视角 / 不懂 → 答 + **patch stages/03-how.md** + 抛新反问
   - 明确推进信号 → 加载 `skills/stage-origin/SKILL.md`,自然过渡

## 注意事项

- **克制深入** —— 这是 How 与 Deep 分工的红线。遇到想展开的细节,用一句"Deep 阶段会精确剖析"过渡
- **demo 不要追求完整** —— 30~80 行,只为手感,不要写产品级代码
- **三个误解不是 FAQ** —— 是戳破读者**会产生**的错觉,不是回答用户问题
- **约束回扣必须用约束编号** —— C1, C5 而不是"性能约束"
- **三件事只引 3 个** —— 第 4、5 个组件留给 §2~§7,§0 别贪多

---

## 反模式

1. **塞入源码细节** —— 那是 Deep 的事
2. **demo 写成完整产品** —— 30~80 行就够,不要超
3. **三个误解写成 FAQ** —— FAQ 是回答问题,误解是戳破错觉
4. **不交叉引用约束编号** —— 散了脊梁
5. **不呼应灵魂问题** —— 失去衔接
6. **§0 三件事铺开成 5~7 个** —— 三件事是纲领性结论,3 个就是 3 个;多了用户记不住,少了又撑不起后面 §2~§7 的细节挂载
7. **没有顶部约束速查表 / 没有 §1 概览图 / 没有 §X 序号** —— 这三件事是 stage 文档"骨架完整"的硬指标,缺一就散。详见入口 SKILL.md《Stage 产物结构纲领》

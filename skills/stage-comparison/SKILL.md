# atlas / stage-comparison

## 阶段目标

把 <主题> 的设计放到**整个设计空间**里看——通过与其他设计的横向对比,让读者意识到 <主题> 的设计选择不是必然,而是特定约束组合下的局部最优。

**核心机制:Comparison Hook** ——位置精确选在 Deep 阶段之后、Synthesis 之前。这时用户对约束 C1...Cn 已经熟了,看到候选清单时能立刻判断"我想看哪个对比"。

---

## 执行流程

### 第一步:调用 progress-tracker 标记阶段开始

标记当前阶段为 "Comparison"。

### 第二步:**渐进式对齐**(必做,一问一答到收敛 —— 见入口 SKILL.md「Stage 开场对齐纪律」)

Comparison 阶段的「候选选择」(下面第四步)本身也是 alignment,但**生成候选前的 scope alignment** 仍然必要 —— 否则 Claude 推荐的候选可能完全不符合用户兴趣。

**第一个问题(最高层方向性)** 示例:

> "在我列对比候选之前 —— 你想要 **同领域深度对比**(对比同类技术的不同实现选择,如 ptmalloc vs jemalloc vs tcmalloc)还是 **跨领域同构对比**(用户态 vs 内核 / 中间件 vs 数据库类似机制)?"

根据回答**自然展开下一个**。可能的展开链:
- 答同领域 → "你已经听说过哪些'竞品'?Claude 优先围绕这些 + 补全你没提到的"
- 答跨领域 → "想跨到哪个领域?(内核 / 数据库 / 编程语言运行时 / 分布式)"
- 用户报候选 → "对比维度优先哪几个?(性能 / 设计哲学 / 历史 / 安全 / ABI 差异)"
- 用户答 → "候选数量:1~2 个深入,还是 3~5 个扫描?"
- 用户答"够了" → **进入"追加 reconfirm"**(不要立刻进生成候选;按入口 SKILL.md「追加 reconfirm」明确问"我已经清楚目标了 —— 你还有什么问题要追加吗?如果没有我就去列对比候选 + 写 stages/06-comparison/06-comparison.md 了")

**严禁一次列 5 个**;**严禁 Claude 觉得清楚了就直接生成候选清单**。收敛信号:用户的对比域 / 候选清单 / 维度优先 / 数量偏好都已明确,**且用户明确说"没有追加,可以写"**。

### 第三步:生成 4~6 个对比候选

读取 stages/05-deep/05-deep.md 中出现的关键设计决策和约束,**自动生成对比候选**。

每个候选必须满足:

1. **绑定一个明确的对比维度**(如"元数据外置 vs 内联"、"per-CPU vs per-thread")
2. **绑定一个核心约束编号**(如"挑战 C3+C5"、"消解 C5")
3. **候选之间维度尽量不重复**——避免同维度多个候选

例如 chunk(malloc)的候选清单:

| 候选 | 对比维度 | 突出约束 |
|------|---------|---------|
| jemalloc | 元数据外置 vs 内联 | C3+C5 |
| tcmalloc | per-CPU vs per-thread tcache | C4 |
| Go runtime malloc | GC 改 ABI vs C ABI 锁死 | C7 |
| Rust 默认分配器 | sized dealloc vs 反向读 size | C5 |
| Linux slab | 内核固定大小池 vs 用户态通用 | C6 |
| buddy allocator | 二的幂分裂 vs bin 分档 | C1 |

**注意**:候选不要超过 6 个。少而精。

### 第四步:用 ask_user_input_v0 让用户选

```
你已经读完 Deep 阶段,对 <主题> 的设计有了完整理解。

下面是几个对比候选,每个突出了 <主题> 在某个约束维度上的特定选择。
你想看 <主题> 与哪些其他设计的对比?(可多选 1~3 个)

[候选 1: jemalloc(元数据外置 vs 内联,挑战 C3+C5)]
[候选 2: tcmalloc(per-CPU vs per-thread,挑战 C4)]
[候选 3: Go runtime(GC 改 ABI vs C ABI,挑战 C7)]
[候选 4: Rust 默认(sized dealloc 消解 C5)]
[候选 5: Linux slab(内核固定池 vs 用户态通用,挑战 C6)]
[候选 6: buddy allocator(二的幂分裂 vs bin 分档,挑战 C1)]
[都不要,跳过对比直接进 Synthesis]
```

用 multi_select 类型。

### 第五步:对比深度分档(自动决定)

根据用户选了几个,自动决定每个对比的写作深度:

- **选 0 个**(选了"都不要")→ **跳过本阶段**,直接调度到 stage-synthesis;调用 progress-tracker 把"用户跳过 Comparison"记录到 PROGRESS.md
- **选 1 个** → 写成 mini 章节(~400 字 + ASCII 图 + 代码片段)
- **选 2~3 个** → 段落级(每个 ~150~200 字)
- **选 >3 个** → 在对话中提示:
  ```
  你选了 X 个对比对象。超过 3 个会稀释每个对比的深度,建议聚焦 2~3 个最关键的。
  你想:
  [收缩到 2~3 个(让我重选)]
  [保持选择,接受较浅的对比]
  ```
  然后用 ask_user_input_v0 给两个按钮

### 第六步:写对照篇

输出到 `stages/06-comparison/06-comparison.md`:

```markdown
# 阶段 6:<主题> 与其他设计的对照

## 对照 N:<主题> vs <对比对象 1>

### 对比维度
[一句话点明]

### <对比对象> 的核心做法
[简述其设计,不是该对象的完整剖析]

### 在五元组上的差异
| | <主题> | <对比对象> |
|---|---|---|
| 核心抽象 | ... | ... |
| 元数据位置 | ... | ... |
| 处理 Cn 的方式 | ... | ... |
| 代价 | ... | ... |

### 为什么 <对比对象> 这么选
[追溯到约束差异:它面对的约束组合不同 → 选了不同候选]

### 这个对比突出了什么
[一句话:让读者意识到 <主题> 的某个约束选择不是必然]

(对每个用户选中的对比对象重复以上结构)

## 《约束回扣》
[对照篇整体说明:<主题> 的设计不是唯一解,而是特定约束组合下的局部最优。
不同分配器/系统选了不同候选,因为它们面对的约束权重不同。]

## 呼应灵魂问题
[对照视角下,灵魂问题的额外洞察]

例如灵魂问题"chunk 为什么是 16B 不是 8B":
> "通过对照,我们看到 jemalloc 选择把元数据完全外置(0 字节内联),Go 选择把
> 元数据按 size class 反查(0 字节内联)。chunk 选 16B 不是因为这是最优,而
> 是 C 语言 ABI 的历史约束让它无法选 0。"
```

### 第七步:在对话中自然地讲清楚 + 抛反问

不说"Comparison 阶段已写入..."。改说:

```
把对比放在了 atlas-output/.../stages/06-comparison/06-comparison.md,选了 N 个对照对象:<列对象>。
关键洞察:<一句话>。

先拷问一下:<反问,例如"你觉得这几个里哪个跟 <主题> 最像?最不像的是哪个,差在哪?">
```

### 第八步:进入 user-pacing 协议

按 `skills/three-options/SKILL.md`:

- 追问 / 新视角 / 不懂 → 答 + **patch stages/06-comparison/06-comparison.md** + 抛新反问
- 用户说"换个对比对象 / 加一个 / 这个对比维度不准"→ 回到第三步重选 / 局部重写
- 明确推进信号 → 加载 `skills/stage-synthesis/SKILL.md`,自然过渡

---

## 关键设计:Comparison 是可选阶段

如果用户在第三步选「都不要」,**直接跳到 Synthesis**,不写 06-comparison.md。

PROGRESS.md 中标记"用户跳过 Comparison",但不计为失败——这是合法选择。

---

## 反模式

1. **候选清单维度全重复** —— 比如同时给 jemalloc/tcmalloc/mimalloc(都是"现代分配器")
2. **不限对比深度** —— 用户选 5 个就硬写 5 个,稀释成水
3. **对比对象 Claude 自己拍板** —— 必须用户选
4. **Comparison Hook 提前到 Discovery 或 What 阶段问** —— 用户那时没判断力
5. **跳过 Comparison 后不调用 progress-tracker 记录** —— 失去审计痕迹

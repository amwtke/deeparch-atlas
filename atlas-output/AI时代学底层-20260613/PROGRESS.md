- 背景输入（用户提供，作为 What 经验锚 + 后续阶段证据）：
  - 用户 ＝ 重度 AI 使用者 + 部门主管；自评效率提升 4–10×，一人 cover 后端→前端→QA→部署。
  - 反常现象①：程序员水平越低越抵触 AI。
  - 反常现象②：初级用 AI 效率反而下降（净产出疑似为负）。
  - 用户强倾向（待 Why/Deep 压力测试）：越懂底层（Linux 内核/网络/GC/Spring 源码/DB 调优）+ 设计模式 → 越能驾驭 AI。
  - Claude 保留的刀刃：被放大的乘数 ＝「底层知识本身」还是「底层养出的判断力/验证直觉」？（ingredient vs feeder，留 Why/How/Deep 钉）
- 第 1 次追问（公式出处）：用户问『净产出 = 判断力 × 增益』是否有事实依据 → Claude 澄清：是 Claude 搭的压缩框架、非实测定律；底层真实理论 ＝ 技能偏向型技术进步(SBTC) / Brooks 本质 vs 偶然复杂度 / GIGO；AI 编程实测倍率混杂，留 Origin 查证。
- 第 1 次新视角：用户把判断力拆成『技术判断（深度，非背诵）+ 业务判断（需求洞察/共情）』两轴 → Claude 映射到协作回路两端（业务↔规约前端 / 技术↔验证后端），据此解释初级翻负，并据用户「非背诵」初步裁定刀刃。→ 已 patch：新增「判断力成分表」小节
- 第 2 次新视角：用户接住「编排判断」第三轴，并升华为『将难的事情变容易的能力』＝拆分/tasking，指出初级拆不动 → 写不出 AI 能懂的 prompt。→ Claude 把成分表升为三轴（业务/编排/技术）铺满回路（前/中/后段），AI 仅占中段「生成」。→ 已 patch：成分表改三轴。
- 第 1 次产物迭代：用户要求「多举例放大」AI 把错的东西造得完美无瑕。→ Claude 新增「死法图鉴」表（DCL 缺 volatile / ThreadLocal 泄漏 / 函数废索引 / @Transactional 自调用 / 重试无幂等 / 照错 ticket 造），分别挂技术判断后段 / 业务判断前段。→ 已 patch：新增图鉴。
